# Slipstream Brainstorm

## Project Name

Slipstream

## Current Intent

Build a global English-assistance system for macOS that helps with daily English-heavy work before the user's English reaches fully native-level fluency.

The system should support tasks such as:

- Communicating with LSE and other institutions.
- Reading English websites.
- Writing English emails.
- Reading English books, papers, slides, and course materials.
- Understanding screenshots or selected regions from arbitrary apps.
- Turning Chinese intent into natural, accurate English output.

## Initial Product Metaphor

Slipstream should combine two familiar interaction patterns:

- Bob-like global language assistance: select text anywhere, trigger a shortcut, and get translation, explanation, rewriting, or email help.
- Lightweight screen capture: trigger a shortcut, select a screen region, then run OCR and AI processing on the captured content.

The important point is that Slipstream should work globally across the user's actual workflow, not only inside a browser tab or chat page.

## Proposed First-Principles Goal

In any app, when the user encounters an English reading, writing, or communication task, Slipstream should let them quickly capture the relevant context and receive directly usable help.

The core loop:

1. Capture context from the current task.
2. Identify or choose the task mode.
3. Send structured input to an LLM.
4. Return an output that is immediately useful.
5. Optionally save the result, revise it, or continue the conversation.

## Core Product Definition

Slipstream is not a plain Chinese-English translator.

The core problem is that the user often reads English materials, browses English websites, and writes emails to university administrative staff. Literal translation is not enough, because some English can cause real misunderstanding when names, terms, or institutional context are mistranslated.

Slipstream should explain:

- Proper nouns: people, places, institutions, abbreviations, course names, departments, programs, forms, and policy names.
- Specialist terms: administrative phrases, course names, academic systems, policy wording, forms, portals, and deadlines.

The default explanation generated by the LLM should do two things:

1. Translate the passage into Chinese.
2. Explain proper nouns and specialist terms.

Slipstream should not present broad "vibe" or cultural judgment as a main output section. If needed, practical notes can be folded into the term explanation, where the claim is tied to a specific word or phrase.

### Core Interactions

#### Clipboard Explanation

```text
copy English text
-> press Option + C
-> floating panel appears
-> LLM explanation appears
```

Option + C is the first default shortcut for clipboard explanation. It must be user-configurable because it may conflict with other apps.

#### Screenshot Explanation

```text
press F2
-> select screen region
-> local OCR recognizes English
-> floating panel appears
-> LLM explanation appears
```

No separate app window should be required for the normal path. The result should appear as a small floating desktop panel.

### Term Memory

Slipstream should store useful terms locally:

- Specialist vocabulary.
- Proper nouns.
- Institutional abbreviations.
- Frequently recurring phrases.
- Terms that the user has asked about before.

Use:

- Link repeated terms across future explanations.
- Show prior explanations when the same term appears again.
- Count frequency.
- Support periodic review.

Term saving should be manual in V0. Automatic extraction can come later.

### Local History

Slipstream should store explanation history locally by default:

- Captured text.
- OCR text.
- Explanation result.
- Provider/model used.
- Timestamp.
- Saved terms.

Screenshots can also be stored locally if useful, but local text history matters more than image history.

### Default Prompt Contract

For any captured English text, the LLM should return a compact explanation in Chinese with this shape:

```text
1. 中文意思
2. 专有名词 / 缩写 / 机构 / 课程名
```

This is a fixed contract, not a free-form chat prompt.

For email or administrative text, the prompt should be especially careful about:

- Deadlines.
- Required documents.
- Whether the user needs to reply.

These should be included only when they are anchored to concrete terms or explicit wording.

## Early Scope

The first version should be a macOS desktop app, not a web app.

Reasons:

- The user's pain points happen across Browser, Mail, PDF readers, Obsidian, Word, slides, and websites.
- A global shortcut and floating window are more important than a full-screen chat UI.
- Screenshot plus OCR is a central interaction, so native macOS integration matters.

## Candidate MVP Features

- Menu bar app.
- F2 screenshot-region capture.
- Local OCR through macOS Vision.
- Clipboard-based explanation through Option + C.
- Floating explanation panel.
- LLM explanation contract:
  - Chinese translation.
  - Proper noun and specialist-term explanation.
- Configurable LLM backend.
- Simple API key setup.
- Local explanation history.
- Manual term saving.

## Candidate Second-Stage Features

- Global selected-text processing.
- History.
- Local term memory.
- LSE / university communication templates.
- Export selected outputs into Obsidian notes.
- Review schedule for saved terms.

## Input-First Plan

The project should still start from the input side, but V0 should include the smallest useful explanation loop.

The first milestone is not "answer everything with AI." It is:

1. Let the user capture useful context quickly.
2. Convert that context into clean text.
3. Send it to the chosen LLM backend.
4. Show a floating explanation panel.

### Input Principles

- Capture should be faster than opening a browser-based AI chat page.
- The user should not need to think about prompt writing during capture.
- The app should work across Browser, Mail, PDF readers, Obsidian, Word, and slide decks.
- OCR text may be editable from the panel, but the normal path should auto-explain without forcing a confirmation step.
- Input capture should remain useful even before the full AI pipeline is mature.

### Input Sources

#### Manual Text Input

The simplest starting point.

- Open Slipstream command window.
- Paste or type text.
- Choose a task mode later.

This is technically easy and useful for early prompt/output testing.

#### Clipboard Input

Use the current clipboard as the input.

- User copies text from any app.
- Presses a shortcut to be chosen.
- App reads the clipboard and starts explanation.

This avoids hard global-selection integration in the first version.

#### Screenshot Region Input

The most important long-term input.

- User presses a shortcut.
- App enters region selection mode.
- User draws a rectangle around website text, email text, PDF text, course slides, or book content.
- App runs local OCR.
- OCR result appears in an editable text box.

This matches the lightweight capture interaction the user wants.

#### Selected Text Input

Useful, but can be treated as a later input source.

- User selects text in another app.
- Presses a shortcut.
- Slipstream captures selected text if possible.

This may require Accessibility permissions and app-specific handling, so it should not block the first prototype.

### Input MVP

The first implementation target should probably be:

1. Menu bar app.
2. F2 starts screenshot-region capture.
3. Local OCR extracts text from the screenshot.
4. Clipboard shortcut reads copied English text.
5. LLM backend generates the default explanation.
6. Floating panel shows the result.
7. Panel allows copying, editing input, or retrying.

At this stage, output can be minimal but real. The important deliverable is the fast capture-to-explanation loop.

### Input Pipeline Draft

```text
trigger
-> choose input source
-> capture raw input
-> normalize input
-> call selected LLM backend
-> show floating explanation panel
```

For screenshot input:

```text
F2
-> region selection overlay
-> screenshot image
-> Apple Vision OCR
-> OCR text cleanup
-> LLM explanation
-> floating panel
```

For clipboard input:

```text
clipboard shortcut
-> read clipboard
-> LLM explanation
-> floating panel
```

### Input Data Shape Draft

```json
{
  "sourceType": "manual | clipboard | screenshot | selectedText",
  "rawText": "...",
  "cleanedText": "...",
  "sourceApp": "optional app name",
  "capturedAt": "timestamp",
  "languageHint": "en | zh | mixed | unknown",
  "metadata": {
    "ocrConfidence": "optional",
    "screenshotPath": "optional"
  }
}
```

### Input Questions To Settle

- Should screenshot-region mode have its own shortcut, or live inside the main command window?
- Should OCR automatically run after screenshot selection, or should the user click "Recognize"?
- Should Slipstream save screenshot images, or only keep extracted text?
- Should OCR results be auto-cleaned aggressively, or should raw OCR text be preserved?
- Should the first app support selected-text capture, or postpone it until after clipboard and screenshot work?

### Input Interaction Draft

Start with two entry points only:

1. Main input panel.
2. Screenshot OCR capture.

Do not start with global selected-text capture. It is useful, but it adds Accessibility permissions and app-specific edge cases. Clipboard and screenshot cover most real use cases with less risk.

#### Entry Point 1: Main Input Panel

Trigger:

- Menu bar click.
- Optional secondary shortcut later.

Behavior:

- Opens a compact floating input window.
- If clipboard contains text, show a "Use Clipboard" action.
- User can paste or type directly.
- Text area is the center of the interface.
- The next-stage action can be disabled until there is text.

Initial controls:

- Use Clipboard.
- Clear.
- Continue.

Skipped for first pass:

- Mode selection.
- Chat history.
- Auto task detection.
- Full conversation UI.

Reason:

- The first useful thing is getting text into Slipstream quickly and cleanly.

#### Entry Point 2: Screenshot OCR Capture

Trigger:

- F2 global shortcut.
- Optional button inside the main input panel later.

Behavior:

- User presses F2 and immediately enters region selection mode.
- User draws a screen region.
- App captures the selected region.
- OCR runs automatically.
- OCR text opens in the same input preview window.
- User can edit before continuing.

Initial controls after OCR:

- Re-capture.
- Copy OCR Text.
- Continue.

Skipped for first pass:

- Saving screenshot history.
- Image annotation.
- Multi-region capture.
- Table reconstruction.
- Layout-aware document parsing.

Reason:

- The first pass only needs to turn visible text into editable text.

### Input Defaults

- F2 is the default global shortcut for screenshot-region capture.
- Option + C is the default global shortcut for clipboard explanation.
- Both F2 and Option + C must be editable in Settings.
- Screenshot OCR should run automatically after region selection.
- OCR output should be editable from the floating panel, but normal use should auto-explain first.
- Store text history locally by default.
- Preserve raw OCR text internally during the session, but show cleaned text to the user.
- Postpone selected-text capture until clipboard and screenshot input feel solid.

### Visual Direction

- Lightweight capture-tool feel.
- Compact floating panel.
- Fast interaction, low chrome, minimal distraction.
- Explanation result should be visually dominant.
- Original source text should be hidden by default, not shown as a main section.
- Dark mode must be readable.
- Normal use should not require opening a full app window.
- UI should stay close to: input/source text, result area, and three or four buttons.
- Avoid documenting or copying any specific existing product's name, layout, or branding.

### Floating Panel Placement

- After screenshot capture, place the panel near the selected region without covering it when possible.
- After clipboard explanation, place the panel near the mouse on the current screen.
- Fallback position: top-right of the active screen.
- Remember the user's last moved panel position later, not required for V0.

### Error States

Errors should be normal user-facing states, not raw technical dumps.

- No API key: show "需要先添加 API key" with a Settings button.
- Empty clipboard: show "剪贴板里没有可解释的文本".
- OCR failed: show "没有识别到清晰文字" with Re-capture.
- Model timeout: show "模型响应超时" with Retry.
- Network/provider error: show a short provider message and Retry.
- Shortcut conflict: show the conflict and link to shortcut settings.
- If Option + C cannot be registered, keep the app usable and guide the user to change the shortcut in Settings.

### V0 Panel Buttons

The floating explanation panel should start with only four actions:

- Copy Explanation.
- Save Term.
- Retry.
- Settings.

Do not add chat, history browsing, annotation, or mode switching to the V0 panel.

### Performance Target

- Copy text to explanation panel appearing should feel under 2 seconds, excluding unavoidable LLM latency.
- Screenshot selection to OCR text should feel immediate for normal website/email/PDF text.
- Slow model calls should show a lightweight loading state in the floating panel.

### Provider Setup

The model setup should be preset-first:

- User chooses a provider.
- App fills base URL, API format, and suggested model names.
- User only pastes an API key for normal setup.

Initial provider presets:

- DeepSeek API.
- OpenAI-compatible custom provider.
- Ollama local provider.

Provider config should support:

- Base URL.
- Model name.
- API key when needed.
- OpenAI-compatible request format where possible.

Full custom provider editing can exist, but it should be behind an advanced path.

### Input V0 Success Criteria

Input V0 is successful if the user can:

1. Copy English text, press a shortcut, and see a floating explanation.
2. Press F2, select a screen region, and see a floating explanation after OCR.
3. Copy the explanation or retry.
4. Open a simple settings path and add an API key.
5. Save a useful term manually.

Term memory, review, selected-text capture, and email-specific drafting can wait.
Local history should be included, but only local storage is required.

## Candidate Modes

### Reading Explanation

Input:

- English paragraph, screenshot OCR text, or selected text.

Output:

- Chinese explanation.
- Key English phrases.
- Important assumptions or hidden requirements.
- Optional short summary.

### Email Assistant

Input:

- Chinese intent.
- Optional recipient/context.
- Optional tone choice.

Output:

- Ready-to-send English email.
- Subject line.
- Optional shorter or more formal version.
- Explanation of sensitive wording when needed.

### English Polishing

Input:

- User-written English.

Output:

- More natural revised version.
- Optional line-by-line explanation of corrections.
- Optional tone variants.

### Website / Admin Page Understanding

Input:

- Screenshot OCR text, copied web text, or page snippet.

Output:

- What the page says.
- What the user needs to do.
- Deadlines, required materials, risks, and next actions.
- Draft email if clarification is needed.

### Course / Book / Slide Understanding

Input:

- Paragraphs, screenshots, lecture slides, or book excerpts.

Output:

- Chinese explanation.
- English terminology.
- Concept map or outline.
- Review questions.
- Optional Obsidian-ready note.

## Open Design Questions

### Input

- Should the first shortcut open a command panel, or should it immediately process selected text?
- Should screenshot mode be a separate shortcut from text mode?
- Should Slipstream auto-detect the task type, or should the user choose a mode manually?
- How much surrounding context should be stored or reused?

### Output

- Should output be compact by default, with expandable details?
- Should every result have one-click copy?
- Should email mode output only the final email first, or include explanation by default?
- Should course-material mode support export to Obsidian from day one?

### Pipeline

- What is the exact input schema sent to the model?
- Should OCR text be cleaned before sending to the model?
- Should every task mode have a fixed prompt template?
- Should the app keep task history locally?
- Should API keys be stored in Keychain?

### Product Boundary

- Is Slipstream primarily an English assistant, or a broader study/work copilot?
- Should it avoid becoming another general chat app?
- Which workflows must be instant, and which can tolerate a larger window?

## Tentative Technical Direction

- Electron + React for the main app shell.
- Thin macOS native bridge for Vision OCR and global shortcuts if Electron cannot cover them cleanly.
- Menu bar resident app.
- F2 screenshot capture shortcut.
- Option + C clipboard explanation shortcut.
- Settings screen must allow changing both global shortcuts.
- Floating result panel, not a normal document window.
- Apple Vision for OCR.
- DeepSeek API as one initial LLM backend.
- Ollama as a possible local backend.
- Backend adapter should support OpenAI-compatible APIs where possible.
- Provider presets should prefill URL, API format, and model names so normal setup only needs an API key.
- Local prompt templates.
- Local explanation history.
- Manual term saving first; automatic extraction later.
- Keychain for API key storage.
- Simple setup: user should be able to paste an API key and start using it.
- Import/export provider config later, similar in spirit to simple model-switcher tools, but not required for V0.
